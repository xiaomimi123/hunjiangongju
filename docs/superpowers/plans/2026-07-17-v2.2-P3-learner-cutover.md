# v2.2 P3 学员端 + 切换上线 实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** 完成 v2.2 换产品的最后一段：(1) 学员端新产品——**浏览下载运营成片库 + 学员自助选框架发起生成**；(2) 仪表盘数据源换成生成/拆解统计；(3) **切换上线**——拆掉旧混剪（表/页/接口/worker），保留学员账号，品牌沿用「东方文澜」。

**Architecture:** 沿用 monorepo + P0/P1/P2 成果（AI 适配层、generation 队列、拆解/生成流水线、`/admin/*`）。P3 新增学员自助生成（复用 P1 生成流水线，学员任务 `autoRender` 自动串联 render+qc，无运营手工门禁）、成片库（运营发布 EXPORTED 成片）、重建学员端页面；最后**破坏性拆旧**（DROP 旧混剪表 + 删旧代码），迁移在切换部署时于线上执行（保留 User 表）。

**Tech Stack:** TypeScript、Prisma、BullMQ、Next.js App Router、vitest。

**已确认决策：** 学员端「两者都要」（成片库浏览下载 + 学员自助生成）；切换保留学员账号、删旧混剪内容；品牌沿用「东方文澜」。

**参考：** `docs/superpowers/specs/2026-07-16-video-deconstruct-generate-v2.2-design.md`（§8 学员页、§12 切换策略）。

## Global Constraints

- **拆旧前保持增量**：Task 1–8 全部增量新增/改，`next build`+`vitest` 始终通过，旧混剪与新产品并存；**Task 9（拆旧）才做破坏性删除**——先建好并验证新学员端，再删旧。
- **保留学员账号**：拆旧 DROP `materials/material_tags/tag_categories/segment_tags/scripts/script_segments/tasks/task_segments/task_status_logs/qc_reports/access_keys` 及对应模型/页/接口/worker job/旧 `pipeline` 队列；**绝不删 `users`**。User 模型移除 `accessKeys`/`tasks` 关系。
- **学员自助生成**：学员用**已发布**框架 + subject 发起；`generationTask.autoRender=true` 时 worker 自动串联（align→render-visuals，render-video→run-gen-qc），学员无需 confirm；运营路径不变（autoRender=false，保留 edit/confirm 门禁）。学员任务 `createdBy=学员id`，各接口按 createdBy 隔离。
- **成片库**：运营把 EXPORTED 生成任务 `published=true`，学员成片库只见已发布成片，可下载。
- **框架库对学员**：只见 `published=true` 的框架。运营在 `/admin/frameworks` 发布/取消。
- 迁移用 `migrate deploy` + 手写 SQL；两容器 `prisma generate`；构建 `NODE_ENV=production`；web 任务跑生产构建门禁后恢复 dev；worker 改动 restart。
- 品牌沿用「东方文澜」（logo/名称/邮件签名不变）。

## P3 验收标准
1. 学员自助生成：学员选已发布框架 + subject → autoRender 自动串联 → EXPORTED → 下载，全在学员端完成。
2. 成片库：运营发布 EXPORTED 成片 → 学员端可浏览下载。
3. 学员端四页（首页/框架库/成片库/我的作品+我的）移动端可用；截图确认。
4. 仪表盘数据源为生成/拆解统计；侧栏无旧混剪入口。
5. 拆旧后：旧混剪表/页/接口/worker 全部移除，`users` 保留；web+worker tsc 干净、`npm test` 全绿、生产构建 EXIT 0。
6. `deploy/CUTOVER.md` 完整可执行（保留学员账号、删旧混剪内容、启用真 key）。
